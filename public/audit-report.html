Gaps Críticos

1. Diagrama visual del árbol OST
La vista "tarjetas" muestra oportunidades en un grid plano. Falta el diagrama jerárquico Outcome → Oportunidades → Hipótesis → Experimentos que es central a la metodología de Teresa Torres.
ESFUERZO: ALTO
2. RLS Policies con enforcement de roles
Las policies actuales permiten a cualquier usuario autenticado escribir. Falta enforcement de que viewers solo pueden leer, y que solo miembros del proyecto acceden a sus datos.
ESFUERZO: MEDIO
3. Realtime subscriptions
Cuando otro usuario hace cambios, no se reflejan en tiempo real. Solo se actualizan con refetch manual. Esto limita la experiencia colaborativa.
ESFUERZO: MEDIO
4. Validación Zod
Sin validación server-side de datos. Un usuario podría insertar datos inválidos via API directa (ej: cita sin source, experiment sin success_criterion).
ESFUERZO: MEDIO
5. "Aplicar sugerencia" en AI Evaluation
El botón existe pero no hace nada más que un toast. Debería parsear la sugerencia de Claude y aplicar los cambios al OST.
ESFUERZO: MEDIO
6. Contraste en modo claro
Varios textos son invisibles o de muy bajo contraste en light mode. Fix en progreso.
ESFUERZO: BAJO

0. Modelo de Dominio OST

Project (contexto organizacional — multi-OST)
  └── Oportunidad (problema / necesidad del usuario)
        ├── Evidencia             → citas, hechos u observaciones que la sustentan
        ├── Hipótesis             → supuesto falseable sobre cómo resolverla
        │     └── Experimento     → prueba diseñada (type, success_criterion, effort, impact)
        └── Evaluación IA         → análisis Claude + conversación de refinamiento
              └── Priorización    → top-3 experimentos (menor esfuerzo, mayor impacto)
EntidadPadreSemántica
opportunitiesprojectsUna oportunidad pertenece a un proyecto
opportunity_evidenceopportunitiesJustifica la oportunidad con datos reales
hypothesesopportunitiesSupuesto falseable sobre cómo resolver la oportunidad
experimentshypothesesPrueba diseñada con criterio de éxito, esfuerzo e impacto
ai_evaluationsopportunitiesEvaluación Claude — audit trail inmutable
ai_conversation_messagesai_evaluationsHilo de refinamiento post-evaluación con Claude
project_membersprojects + auth.usersRol del usuario en el proyecto

1. Schema de Persistencia (SQL)

CREATE TABLE projects (
  id UUID PK, name TEXT NOT NULL, description TEXT,
  created_by UUID REFERENCES auth.users(id),
  created_at TIMESTAMPTZ DEFAULT now(), updated_at TIMESTAMPTZ DEFAULT now()
);

CREATE TABLE project_members (
  project_id UUID FK, user_id UUID FK,
  role TEXT CHECK (role IN ('admin', 'usuario', 'viewer')),
  invited_at TIMESTAMPTZ DEFAULT now(), PRIMARY KEY (project_id, user_id)
);

CREATE TABLE business_context (
  id UUID PK, project_id UUID FK, content TEXT NOT NULL,
  updated_at TIMESTAMPTZ DEFAULT now()
);

CREATE TABLE opportunities (
  id UUID PK, project_id UUID FK, name TEXT NOT NULL,
  description TEXT, outcome TEXT, archived BOOLEAN DEFAULT false,
  created_at TIMESTAMPTZ, updated_at TIMESTAMPTZ
);

CREATE TABLE opportunity_evidence (
  id UUID PK, opportunity_id UUID FK,
  type TEXT CHECK (type IN ('cita','hecho','observacion')),
  content TEXT NOT NULL, source TEXT, -- obligatorio si type='cita'
  created_at TIMESTAMPTZ
);

CREATE TABLE hypotheses (
  id UUID PK, opportunity_id UUID FK, description TEXT NOT NULL,
  status TEXT CHECK (status IN ('to do','en curso','terminada')),
  result TEXT, created_at TIMESTAMPTZ, updated_at TIMESTAMPTZ
);

CREATE TABLE experiments (
  id UUID PK, hypothesis_id UUID FK,
  type TEXT CHECK (type IN ('entrevista','prototipo','smoke_test','prueba_usabilidad','otro')),
  description TEXT NOT NULL, success_criterion TEXT NOT NULL,
  effort TEXT CHECK (effort IN ('bajo','medio','alto')),
  impact TEXT CHECK (impact IN ('bajo','medio','alto')),
  status TEXT CHECK (status IN ('to do','en curso','terminada')),
  result TEXT, created_at TIMESTAMPTZ, updated_at TIMESTAMPTZ
);

CREATE TABLE ai_evaluations (
  id UUID PK, opportunity_id UUID FK,
  prompt_snapshot TEXT NOT NULL, evaluation_text TEXT NOT NULL,
  created_at TIMESTAMPTZ
);

CREATE TABLE ai_conversation_messages (
  id UUID PK, evaluation_id UUID FK,
  role TEXT CHECK (role IN ('user','assistant')),
  content TEXT NOT NULL, created_at TIMESTAMPTZ
);

-- RLS: todas las tablas con ROW LEVEL SECURITY habilitado
-- Helper: user_role_in_project(p_id) → retorna el rol del usuario
-- Policy: members can read, editors (admin/usuario) can write

4. Contratos TypeScript + Zod

// Roles
ProjectRoleSchema = z.enum(['admin', 'usuario', 'viewer'])

// Evidencia — source obligatorio para cita
OpportunityEvidenceSchema = z.object({
  type: z.enum(['cita', 'hecho', 'observacion']),
  content: z.string().min(1),
  source: z.string().optional(),
}).superRefine((data, ctx) => {
  if (data.type === 'cita' && !data.source)
    ctx.addIssue({ path: ['source'], message: 'source obligatorio para citas' })
})

// Experiment
ExperimentSchema = z.object({
  type: z.enum(['entrevista','prototipo','smoke_test','prueba_usabilidad','otro']),
  description: z.string().min(1),
  success_criterion: z.string().min(1),
  effort: z.enum(['bajo','medio','alto']),
  impact: z.enum(['bajo','medio','alto']),
  status: z.enum(['to do','en curso','terminada']),
  result: z.string().optional(),
})

// Priority Score
SCORE = { bajo: 1, medio: 2, alto: 3 }
priorityScore(exp) = SCORE[impact] / SCORE[effort]
