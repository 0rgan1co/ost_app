Brechas críticas

fix-ai-proxy-schema-v3

La edge function ai-proxy usa hypothesis_id / DBHypothesis del schema v2. El front-end hace tiempo migró a assumption_id + solutions. Claude recibe un prompt que no refleja el árbol real.

Crítica14 tasksSin dependencia

consolidate-anthropic-access

Coexisten (1) cliente directo con dangerouslyAllowBrowser: true y VITE_ANTHROPIC_API_KEY expuesta al bundle, y (2) edge function ai-proxy. Riesgo de seguridad: la API key viaja al browser.

Crítica16 tasksDepende: fix-ai-proxy

add-supabase-retry-screen

REQ-OST-008 pedía pantalla bloqueante con retry al fallar Supabase en bootstrap. Nunca llegó al código: hoy con red inestable la app queda en spinner indefinido.

Alta11 tasksSin dependencia

add-update-debounce

REQ-OST-010 pedía debounce de 500 ms en UPDATEs. Hoy cada keystroke dispara un round-trip — genera flicker vía eventos Realtime que llegan con valores a-medio-tipear.

06Decisiones de migración

Opción elegida: C (mixta)

1) Baseline honesto en specs/ reflejando código real · 2) changes por brechas críticas · 3) changes por features huérfanas consolidadas.

Estilo de Requirements

A1 · Tipo EARS implícito en el phrasing (SHALL / While / If then / When), no marcado explícitamente.

Trazabilidad al spec original

B1 · El ID REQ-OST-XXX aparece sólo en el bloque ## Purpose de cada capability como referencia, no por requirement.

Features huérfanas

C1 · Si están consolidadas en código, entran al baseline. Si son menos estables, pasan por changes/document-* y se archivan.
